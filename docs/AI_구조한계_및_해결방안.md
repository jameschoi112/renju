# AI 구조적 한계와 해결 방안

## 한계 요약

| 한계 | 원인 | 고성능 툴과의 격차 | 비고 |
|------|------|-------------------|------|
| **언어/런타임** | JS는 C/Rust 대비 5~20배 느림 | 결정적 | 미해결 |
| **매 노드 전체 평가** | `evalBoard(b)` 호출 시 보드 전체 O(N²) 스캔 | 노드당 비용 과대 | ✅ 증분 평가로 완료 |
| **단일 스레드 검색** | 워커 1개만 사용 | 멀티코어 미활용 | ✅ 2워커 병렬 + depth 선택 |
| **알고리즘** | MTD(f), null-move 등 미적용 | 탐색 효율 한계 | ✅ MTD(f) 적용 |

---

## 해결 방안 (우선순위)

### 1. 증분 평가 (Incremental Eval) ✅ 완료
- **구현**: `eval.js`의 `deltaEval(b, r, c, color)`로 한 수가 관여하는 4방향만 재계산. minimax에서 `baseScore + deltaEval(...)` 유지.
- **효과**: 노드당 평가 비용 O(1) 수준. 루트에서만 `evalBoard(b)` 호출.

### 2. WebAssembly (WASM)
- **목표**: 검색/평가 코어를 C·Rust로 작성 후 WASM으로 컴파일, JS에서 호출.
- **효과**: 동일 로직만으로도 **수 배~수십 배** 속도 향상 가능.
- **방법**: Rust로 minimax + 평가 이식 → `wasm-pack` 등으로 빌드 → Worker에서 WASM 모듈 로드.
- **비용**: 이식·디버깅 공수 큼. 기존 오프닝/VCF/VCT 로직 연동 필요.

### 3. 다중 워커 병렬 탐색 ✅ 완료
- **구현**: `game.js`에서 `NUM_AI_WORKERS = 2`, 각 워커에 `timeLimitMs = AI_TIME_MS/2`, `returnDepth: true`로 호출. 두 결과 중 `depthReached`가 큰 쪽을 선택.
- **효과**: 2코어 활용, 같은 실시간 안에 두 번의 독립 탐색 후 더 깊은 결과 사용. (TT 미공유로 히트율은 워커별.)

### 4. 알고리즘 보강
- **MTD(f)** ✅ 완료: 반복적 심화 시 널 윈도우 탐색을 반복해 정확한 점수로 수렴. `rootSearchWithWindow` + `mtdf()`, TT에 Lower/Upper 플래그 저장.
- **Null-move pruning**: (미적용) 수를 두지 않고 한 번 패스한 뒤 감소 깊이로 탐색해 베타 컷. (렌주는 전술적이라 R=2, 제한적 적용.)
- **Late Move Reduction 강화**: LMR 조건/감소량 튜닝으로 불필요한 가지 축소.

### 5. 데이터 구조
- **비트보드**: 15×15를 비트로 표현해 한 방향 라인 체크를 비트 연산으로 (JS에서도 부분 적용 가능). (미적용)
- **TT 확대** ✅ 완료: `hash.js`에서 1M → 4M 엔트리(1<<22). 메모리 대비 히트율·품질 개선.

---

## 결론
- **적용 완료**: 증분 평가, MTD(f), 다중 워커(2개)까지 반영됨. 같은 JS 코드로 강도·탐색 효율·멀티코어 활용이 개선된 상태.
- **장기적**: WASM 이식 시 “고성능 툴만큼은 안 된다”는 격차를 크게 줄일 수 있음.
- **추가 여지**: Null-move(제한적), LMR 강화, 비트보드 등.
